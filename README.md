# PC Assistant Bot

Безопасный Telegram-бот для личного ПК с allowlist-архитектурой. Это не RAT и не скрытый доступ, а прозрачный бот для мониторинга системы, статуса игр, запуска разрешённых приложений и базовых сценариев.

## О проекте

PC Assistant Bot — это Telegram-бот для управления личным ПК в безопасной модели доступа. Бот работает только с заранее разрешёнными пользователями, чатами, приложениями, папками, сценариями и действиями, а его конфигурация вынесена в `.env` и YAML-файлы, что делает поведение прозрачным и управляемым.[1][2]

Основная идея проекта — не давать «полный удалённый доступ», а разрешать только заранее описанные и проверенные действия. Такой подход уменьшает риск случайного или опасного использования и делает проект ближе к personal assistant tool, чем к скрытому remote-control решению.[3]

## Возможности

На текущем этапе проект ориентирован на такие задачи:

- просмотр статуса системы;
- просмотр списка процессов;
- определение игрового статуса по процессам;
- запуск только разрешённых приложений;
- просмотр разрешённых файлов из разрешённых папок;
- запуск базовых сценариев;
- подтверждение чувствительных действий через inline-кнопки;
- централизованная конфигурация через `.env` и YAML.

## Архитектура

Проект построен вокруг allowlist-подхода. Доступ ограничивается на нескольких уровнях: через `AUTHORIZED_USER_IDS` и `AUTHORIZED_CHAT_IDS`, через allowlist приложений, через allowlist папок и расширений файлов, а также через заранее заданные сценарии и команды.[3]

Конфигурация разделена по назначению:

- `.env` — токен бота, авторизация, базовые параметры запуска;
- `data/apps.yaml` — разрешённые приложения и их процессы;
- `data/folders.yaml` — разрешённые корневые папки, лимиты размера и расширений;
- `data/scenarios.yaml` — сценарии из пошаговых действий;
- дополнительные YAML-файлы — игры, лаунчеры, alerts, scripts и другие настройки.[3]

## Установка

1. Создай Telegram-бота через BotFather и получи токен.
2. Скопируй `.env.example` в `.env`.
3. Заполни в `.env` необходимые значения:
   - `BOT_TOKEN`
   - `AUTHORIZED_USER_IDS`
   - `AUTHORIZED_CHAT_IDS`
4. Замени `USERNAME` и пути в YAML-конфигах на реальные значения для своей Windows-системы.
5. Установи зависимости:

```bash
pip install -r requirements.txt
```

6. Запусти бота:

```bash
python bot.py
```

## Конфиги

### Приложения

В `data/apps.yaml` задаются разрешённые приложения, путь к исполняемому файлу, аргументы запуска и имена процессов. Для Windows-путей безопаснее использовать единый стиль записи и внимательно проверять существование файлов до запуска.[3][4]

### Папки и файлы

В `data/folders.yaml` задаются разрешённые корневые папки, максимальный размер файлов и allowlist расширений. В коде важно проверять нормализованные абсолютные пути, а не просто строковые префиксы, чтобы не допустить обход через нестандартные пути или `..`.[3]

### Сценарии

В `data/scenarios.yaml` сценарии описываются как список шагов с типами вроде `open_app`, `set_volume`, `ping_check`, `health_check` и других. Такой конфиг удобен для расширения, но реальные обработчики каждого шага должны быть реализованы отдельно в коде.[3]

### Alerts

В `data/alerts.yaml` могут храниться пороги CPU, RAM, disk, ping, длительности игровых сессий и флаги уведомлений. Сам конфиг уже полезен как основа, но его ценность появляется только после подключения реального фонового мониторинга.[3]

## Безопасность

Проект специально ограничен allowlist-архитектурой: бот не должен выполнять произвольные команды, открывать произвольные пути или запускать неизвестные программы. Все действия должны проходить через заранее описанные конфиги и проверяемые обработчики.[3]

Для файловой части важно:

- работать только внутри разрешённых root-папок;
- проверять расширения по allowlist;
- ограничивать размер файлов;
- нормализовать пути перед проверкой;
- не допускать произвольную передачу пользовательского пути без валидации.[3]

Для приложений и сценариев важно:

- запускать только заранее разрешённые `.exe` и скрипты;
- не выполнять произвольные shell-команды из Telegram;
- подтверждать опасные действия через inline confirm/cancel;
- логировать чувствительные операции.[3]
